📖 概述

Agent.py 实现了一个 基于 LLM 的智能代理（Agent）， 核心设计采用 ReAct 循环（Reasoning + Acting）， 即：代理接收用户输入后，与 LLM 进行多轮对话，LLM 可以按需调用外部工具， 直到 LLM 认为任务完成并返回最终文本回复。

🎯 核心设计理念：Agent 不是单次问答，而是一个 「思考→行动→观察→再思考」 的循环过程， 直到 LLM 不再需要调用工具为止。

📦 依赖模块

json 标准库

用于序列化工具调用参数（json.dumps），将 Python 字典转为 JSON 字符串传递给 LLM。

Message 项目模块

来自 agent.data，是消息数据类，封装了对话中的角色（system / user / assistant / tool）、内容、工具调用信息等字段。

LLMClient 项目模块

来自 agent.llm.base，LLM 客户端的抽象基类（或接口），定义了 chat(msgs, tools) 方法，用于与具体的大模型（如 OpenAI、Claude 等）进行通信。

ToolRegistry 项目模块

来自 agent.tools.registry，工具注册中心，管理所有可用工具的注册、查询和执行。提供 get_all_tools() 和 execute(name, **kwargs) 两个核心方法。

🔍 逐段代码解析

第一段：用户消息入队

self.history.append(Message(role="user", content=user_input))

将用户的输入包装成一个 Message 对象，角色为 "user"，追加到对话历史中。 此后 self.history 至少包含一条用户消息。

第二段：构建发送给 LLM 的消息列表

messages = []
if self.system_prompt:
    messages.append(Message(role="system", content=self.system_prompt))
messages.extend(self.history)

每当向 LLM 发送请求时，都会重新构建消息列表：

  • 如果存在 system_prompt，将其作为第一条消息（角色 system）
  • 然后拼接完整的对话历史（包含用户消息、助手回复、工具调用及结果）

第三段：调用 LLM

chat_response = self.client.chat(
    msgs=messages,
    tools=self.tool_registry.get_all_tools()
)

将消息列表和所有可用工具的定义一起发送给 LLM。get_all_tools() 返回工具定义的列表（JSON Schema 格式）， LLM 据此判断是否需要调用某个工具。

第四段：处理工具调用（分支逻辑）

if chat_response.tool_calls:
    # 步骤 A：记录 assistant 的 tool_calls 消息
    self.history.append(Message(role="assistant", tool_calls=[...]))

    # 步骤 B：逐个执行工具
    for tool_call in chat_response.tool_calls:
        res = self.tool_registry.execute(
            tool_call.name, **tool_call.arguments
        )
        # 步骤 C：工具结果写回 history
        self.history.append(Message(
            role="tool", content=res,
            tool_call_id=tool_call.id
        ))

    # 然后继续 while 循环

当 LLM 返回 tool_calls 时，执行以下三步：

  1. 记录工具调用请求 — 构造一条角色为 assistant 的消息， 其中包含 tool_calls 数组，每个元素有 id、type 和 function（含名称与 JSON 序列化后的参数）。
  2. 执行工具 — 遍历每个 tool_call，调用 tool_registry.execute()， 传入工具名称和参数字典。
  3. 结果回写 — 将工具执行结果包装为角色 tool 的消息， 通过 tool_call_id 与请求关联，追加到 history。

第五段：无工具调用 — 终止循环

else:
    self.history.append(Message(role="assistant", content=chat_response.content))
    return chat_response.content

如果 LLM 没有返回 tool_calls，说明它认为任务已经完成：

  • 将助手的文本回复追加到对话历史
  • 返回文本内容，结束 while True 循环

✨ 设计亮点

🔄 无限循环 + 自然终止

使用 while True 实现持续的工具调用循环，不限制迭代次数， 完全由 LLM 自主决定何时停止。这种设计灵活但有风险，实际生产环境通常会加上 max_iterations 上限。

🧠 完整对话历史

self.history 保留了完整的上下文：用户输入、助手 tool_calls、工具返回结果、助手最终回复。 这使得 LLM 在每一轮都能"看到"之前所有的交互，做出更准确的判断。

🔌 依赖注入 / 松耦合

LLMClient 和 ToolRegistry 都通过构造函数注入，Agent 不依赖任何具体实现。 可以轻松替换不同的 LLM 后端（OpenAI、Claude、本地模型）和工具集。

📐 OpenAI 兼容的消息格式

tool_calls 的数据结构（id、type: "function"、function.name、function.arguments） 与 OpenAI Chat Completions API 保持一致，方便对接各类兼容接口。

⚠️ 注意事项与改进建议

🔴 潜在风险：
  • 无限循环可能导致 token 消耗过大或陷入死循环（建议添加 max_steps 限制）
  • 缺少异常处理——工具执行失败时不会捕获异常，可能导致整个 run 崩溃
  • 缺少流式输出支持——用户需等待完整回复，体验不够实时
💡 改进方向：
  • 添加 max_iterations 参数防止无限循环
  • 用 try/except 包裹工具执行，失败时将错误信息作为 tool 消息返回给 LLM
  • 支持流式（streaming）输出，提升交互体验
  • 考虑添加 reset() 方法清空对话历史
  • 日志记录每次 LLM 调用和工具执行，便于调试
